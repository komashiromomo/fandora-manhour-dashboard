# Fandora 人工時管理系統 — 開發紀錄

> 從一個 Excel 檔到一套即時 BI dashboard 的歷程

---

## 一、為什麼要做這個系統

### 痛點

過去工時管理在 Google Sheets 上散落維護：

| 問題 | 影響 |
|---|---|
| 13 位員工各自一份「個人版工作日誌」+ 主管彙整版本 | 月底結算工時要人工 vlookup 比對 |
| IP × 部門 × 工作項目 × 員工 多維度，但 sheet 只能呈現 2D | 想看「老高與小茉這個 IP 在 2026 Q1 共投入多少工時 / 哪些人」沒辦法即時得到 |
| 工時無法直接對應到費用 | 每個專案的真實成本 = 直接成本 + 分攤管銷，但管銷要怎麼分？ |
| 員工填寫品質參差不齊 | 有人把 IP 名稱填到「工作項目」欄，造成統計失真，但 sheet 無法自動偵測 |

### 系統目標

- **真實工時可視化**：從原始日誌到 dashboard 全自動，免人工結算
- **多維度交叉分析**：IP × 部門 × 員工 × 工作項目 × 月份 任意組合下鑽
- **精算管銷分攤**：把公司每月管銷（人事費 + 房租 + 硬體 + 雜費）按工時佔比攤到每個 IP，得到「真實獲利前的成本」
- **資料品質守門員**：自動偵測員工誤填、低品質記錄並提醒

## 二、開發歷程（時間軸）

### 起點：兩次失敗的 MVP（2026 年 3 月底 – 4 月初）

第一次嘗試做了一個 611 行的單檔 React 應用，部署在 GitHub Pages，能跑但問題堆積：
- Excel 日期欄位被誤判成專案名稱
- 同一員工有 3 種名字（中文 / 英文 / 暱稱），統計時對不上
- 接續列空白會丟資料（員工不會把每行都重複填名字）

第二次重寫想做模組化，但 34 個檔案完全沒進版本控制就直接 deploy 結果整套白屏。**過程中沒任何人能看到 dashboard 跑出來**。

教訓：先小、先穩、先 commit。

### 接手 v3（2026 / 4 / 26 – 5 / 4，9 天）

從第三次重寫的半成品接手。整體進展可分 6 個階段：

---

### 階段一：把資料管路打通（4 / 26 – 4 / 28）

> 「先讓老闆看得到任何一筆資料，再說設計。」

**面對的現實**

接手時發現 v3 是個「殼」：登入頁、頁籤、UI 都做了，但 **真正的資料解析器是空的**。從 Drive 載入按下去什麼都不會發生，但前端不會報錯（靜默失敗）。

**每一道關卡與克服**

| 卡關 | 解決方式 |
|---|---|
| 登入按鈕按了沒反應 | Google 帳號登入的 React 元件接線錯誤，修好按鈕能渲染 |
| 登入後看不到「設定」頁 | 角色判定機制沒接上，所有人都被當一般成員。改成 admin 白名單 + 公司權限表 |
| 同事帳號登入後中文名字讓畫面卡死 | Google 回傳的姓名含「捲毛」這種中文字，前端解碼程序對中文支援有 bug，修好 |
| Drive 上的工作日誌資料夾抓不到內容 | 公司用的是「共用雲端硬碟」（Shared Drive），Drive API 預設不會搜共用硬碟，要明確標示 |
| 載入後每個檔案產 0 筆資料 | Excel 日期欄是 number（45662 代表某日的 Excel 序號），程式把它當字串處理，整列被跳過 |
| 每次 reload 都要重新解析 21 個檔，一次 30 秒 | 加入瀏覽器本機快取，reload 後直接從快取還原；只有主動刷新才重抓 |

**里程碑**：成功解析 21 個個人版日誌、5,385 筆工時記錄。

### 階段二：套上 Fandora 設計風格（4 / 28）

接手時 dashboard 是預設藍色 + Ant Design 標準風格，跟公司品牌完全沒關係。

**做法**：用 Claude Design 工具一起跟著設計師（AI assistant）迭代了一輪「人工時系統 V2」設計：

- 採用公司 CI 配色（cyan #00A4C6 主色 + 橘 / 暖配色）
- 字型統一 Noto Sans TC + Roboto + Fredoka（標題用）
- **改成左側欄導覽**（取代頂端頁籤），分成「分析」「系統」兩組
- Header 加上麵包屑、即時同步指示、品牌標識
- 所有圖表配色換成設計 tokens 控制
- 6 個分頁全部重寫成設計風

**設計系統能力**：所有顏色、間距、半徑、陰影都用 CSS 變數管理，未來換主題色只要改 tokens。

### 階段三：成本分攤從個人月薪改為公司月度管銷（5 / 4）

接手時的成本邏輯是「員工月薪 × 工時佔比 → 專案成本」，但 Fandora 實際提供的「Cost Sheet」不是員工薪資，而是**公司每月管銷費用表**：

| 月份 | 人事費 | 房租場租 | 硬體系統費用 | 雜費 | 總計 |
|---|---|---|---|---|---|
| 2026/04 | $1,000,000 | $100,000 | $40,000 | $60,000 | $1,200,000 |

**重新定義公式**：

```
專案 P 在月份 M 的成本 = 該月公司總管銷 × (P 在該月工時 / 全公司在該月總工時)
專案 P 的總成本 = Σ_所有月份 上述
```

部門、員工、工作項目同樣的算法。所以一個 IP 投入越多工時，分攤到的管銷越多 — 反映真實的資源排擠成本。

**這個公式帶來的能力**：

- 每個 IP 自動算出當期累計成本（依當月公司管銷與全公司工時動態調整）
- 同樣的工時若那個月公司管銷比較重（年終獎金月、新硬體採購月），分攤就比較重 — 反映季節性
- 全公司平均時薪（公司管銷 ÷ 總工時）作為基準，可比較哪些 IP 的「每小時產出成本」高於 / 低於平均

### 階段四：多維度進階分析（5 / 4）

從 5 個分析頁（總覽 / 專案 / 部門 / 人員 / 工時類型）擴充到 **可任意層次下鑽**：

```
專案頁 ──點老高與小茉──▶ 抽屜開啟「老高與小茉」全面分析
         ──點工作項目「商開設計」──▶ 篩成「老高與小茉 × 商開設計」
         ──點員工「余凱紓」──▶ 再篩成「老高與小茉 × 商開設計 × 余凱紓」
         ──點 2026-04──▶ 看到該月每天的詳細記錄
```

5 個維度（IP / 員工 / 部門 / 工作項目 / 月份）可任意組合，**麵包屑**讓你逐層回退。

意義：經營層問「老高與小茉的成本主要是商開部花的，還是視覺設計部花的？」三秒內可以給答案。

### 階段五：資料品質守門員（5 / 3 – 5 / 4）

**問題**：員工填寫工時欄位很自由，常見三類錯誤：

1. **把 IP 名稱填到「工作項目」欄**：例如把「魔物獵人」當成今天做的事情記下來，但「魔物獵人」是 IP（作品名），不是工作（動作）
2. **範例列被當成資料**：日誌前 2 列是給員工示範填法的「範例 1 / 範例 2」，但程式以為是真的工時
3. **範例列的 IP 帶到後面**：範例 2 通常填「老高與小茉」當示意，造成所有沒填 IP 的列被誤算成「老高與小茉」（這個 bug 害我們以為老高佔了 4254 小時，實際只有 69）

**解法**：

- **嚴格日期驗證**：只有 A 欄填了實際日期（4/1、2026/04/01 或 Excel 序號）的列才算資料，header / 範例 / 空白全部跳過
- **IP 誤填偵測 + 警示**：六個分析頁頂部都會顯示橘色警示卡：「⚠ N 個工作項目其實是 IP 名稱，共 X 小時，N 位員工」，點擊可看到「誰寫錯」明細表
- **每週自動健檢**：每 7 天打開 dashboard 第一次，自動跳出 modal 點名本週仍在誤填的員工，可一鍵 snooze 一個月
- **可維護的 IP 清單**：設定頁加「IP 清單管理」，自動偵測 Drive 上實際出現的 IP，發現「不在清單上的新 IP」會列出來，點一下就加入；清單變動所有頁面即時生效

### 階段六：分享、自動化、視覺風格（5 / 4）

**半自動同步**：dashboard 預設每 30 分鐘背景靜默拉一次 Drive 新資料，不打擾使用者；可在右下角設定面板調 15 / 30 / 60 / 240 分鐘或關閉。token 過期會靜默放棄這次（不會跳授權視窗），等使用者下次親自操作再處理。

**可分享連結（最新）**：

dashboard 改成 hash routing，URL 反映當前看到的 view：

```
fandora-manhour-dashboard.vercel.app/#/projects/老高與小茉
fandora-manhour-dashboard.vercel.app/#/projects/老高與小茉/workType:商開設計/employee:余凱紓
```

複製這個連結貼給同事，打開直接落到那個 view。Header 加了「複製連結」按鈕一鍵複製。

**動森島嶼風格**：跟設計工具迭代出第二套 UI 風格（牛皮紙厚邊框、3D 凸起陰影、圓潤 pill 按鈕、四季背景、飄落葉子動畫），可在右下角即時切換「簡潔 ↔ 動森」。同一份資料兩套外觀，氛圍會議用動森，給投資人看用簡潔。

- **三個主色系**（冷 / 暖 / 深色）
- **三段排版密度**（寬鬆 / 平衡 / 密集）

## 五、過程中關鍵的選擇

### 為何不用市售 BI 工具（Looker / Power BI / Tableau）

- 工時源資料是 Google Sheets，欄位格式不統一（有人把 IP 填到工作項目欄、有人填中文有人填暱稱），先要做一層**清理 + 對應 + 容錯**才有意義
- 公司只想要「自己的 dashboard」不想再多一個 SaaS 帳號管理
- 成本分攤公式有特殊邏輯（「公司管銷 × 工時佔比」），需要客製化計算

自製成本：9 天工程，無持續訂閱費。

### 為何用 React SPA + Vercel

- 沒有後端 — 所有資料拉取與計算在瀏覽器內完成，安全（OAuth token 不離開瀏覽器）
- Vercel 自動部署，push 到 main 約 1 分鐘上線
- Google Drive API 直連，沒中間人

### 為何「兩種風格」設計

公司氛圍會議要的視覺感跟對外簡報要的不同。同一份資料、兩種人格，切換零成本。

## 六、下一步路線圖

### 已寫好規格、未開發

**專案損益系統**（規格書已撰寫）

接這個工時系統後，最後一塊拼圖是把 **營收 / 直接成本 / 授權成本 / 行銷費用** 也接進來。當前精算的「分攤管銷」就是損益表的成本端壓軸。預期介面：

```
營收           − 直接成本 = 毛利
毛利           − 授權成本 − 行銷費用 − 分攤管銷 = 營業淨利
營業淨利率     = 營業淨利 / 營收
工時投報       = 營業淨利 / 投入工時 (新指標)
```

每個 IP 一張月度損益走勢、預警「OP 為負」的 IP、跨期比較。

### 待強化

- **權限細分**：目前 admin / member 兩級；未來可加 finance role 只看損益、各部門主管只看自己部門
- **匯出 PDF / Excel**：給董事會月會用
- **手機版**：當前 1280px 以下會堆疊但體驗不夠順
- **資料品質指標卡**：目前只有 IP 誤填警示，可擴充「漏填日數」「異常時數（日 < 4h 或 > 12h）」「連續未填」等
